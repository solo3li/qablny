# ⚙️ تحليل تطبيق **قابلني** (Qablny) - الخادم (Backend)

> خادم قوي ومبني على معمارية متينة لتطبيق Qablny، يعمل ببيئة ASP.NET Core 10 كخادم API مركزي.

---

## 🧱 Stack التقني

| الطبقة | التقنية |
|--------|---------|
| Framework | **.NET 10 (ASP.NET Core Minimal APIs)** |
| Database | **PostgreSQL** + **Entity Framework Core 10** |
| Real-time | **SignalR** للاتصال اللحظي وإدارة الحضور |
| Cache & Queue | **Redis** (مخطط للمطابقة وحفظ التواجد) |
| WebRTC | **LiveKit Server** لإنشاء غرف الصوت والفيديو الحقيقية |
| Storage | **MinIO** (S3 Compatible Object Storage) لرفع الوسائط |
| AI Translation | **LibreTranslate** (محلي عبر Docker للترجمة الفورية) |
| Containerization | **Docker & Docker Compose** للتشغيل والنشر |

---

## 🗂️ بنية الملفات (Single Project Architecture)

المشروع مبني كـ Single Project لتسهيل التطوير السريع مع الحفاظ على فصل الاهتمامات (Separation of Concerns):

```text
backend/
├── Qablny.API.csproj      ← ملف إعدادات المشروع الأساسي
├── Program.cs             ← نقطة البداية وإعداد حقن التبعيات (DI) و Middleware
├── Entities/              ← نماذج قواعد البيانات (Domain Models مثل User, Message, Friend)
├── Enums/                 ← التعدادات (Enums مثل MessageType)
├── Data/                  ← AppDbContext وتهيئة جداول قاعدة البيانات
├── DTOs/                  ← سجلات نقل البيانات (Request/Response Records)
├── Services/              ← منطق الأعمال المعزول (Business Logic)
├── Hubs/                  ← SignalR Hubs للاتصالات اللحظية المستمرة
└── Controllers/           ← نهايات الـ REST APIs الأساسية
```

## 🔌 المكونات الرئيسية

### 1. SignalR Hubs (`ChatHub.cs`)
المسؤول الأول عن الاتصال اللحظي في التطبيق:
- **إدارة التواجد (Presence):** تتبع المستخدمين المتصلين حالياً.
- **توجيه الرسائل (Routing):** إرسال واستقبال الرسائل (`ReceiveMessage`, `MessageSent`).
- **إشارات المكالمات (Signaling):** إرسال إشارات الرنين (`IncomingCall`)، القبول (`CallAccepted`)، الرفض (`CallDeclined`)، والإنهاء (`CallEnded`).
- **تسجيل مدة المكالمات:** يقوم السيرفر بحساب مدة المكالمة منذ بدايتها وحتى استدعاء `EndCall`، ثم ينشئ رسالة من نوع `VoiceCall`، `VideoCall`، أو `MissedCall` ويرسلها فوراً للعميلين لتظهر بالشات وتوثق بالمحادثة.

### 2. خدمات النظام (Services)
- **`AuthService.cs`:** إدارة عمليات تسجيل الدخول وتوليد رموز (JWT) وتشفير كلمات المرور باستخدام خوارزمية BCrypt لحماية البيانات.
- **`MessageService.cs`:** حفظ الرسائل المتبادلة في قاعدة البيانات وإرسالها لخدمة `LibreTranslate` للحصول على ترجمة فورية تلقائية متى لزم الأمر.
- **`LiveKitService.cs`:** الاتصال بمكتبات LiveKit لتوليد توكنز الـ JWT الخاصة بالخادم للعملاء لتمكينهم من الدخول لغرف الصوت والفيديو بأمان.
- **`UserFriendService.cs`:** إدارة العلاقات بين المستخدمين وطلبات الصداقة.
- **`MinioStorageService.cs`:** معالجة ورفع الملفات (الصوت والصور) إلى خادم التخزين MinIO وإرجاع الروابط المباشرة الدائمة.

### 3. قاعدة البيانات (`Data/AppDbContext.cs`)
- تستخدم **Entity Framework Core** لإدارة جداول الـ PostgreSQL.
- تحتوي على الجداول الأساسية: المستخدمين (`Users`)، الرسائل (`Messages`)، الأصدقاء (`Friends`)، الهدايا (`Gifts`) وغيرها من العلاقات المترابطة.

## 🌐 البنية التحتية (Docker Compose)
الخادم مصمم ليعمل بضغطة زر واحدة بالاعتماد الكامل على ملف `docker-compose.yml` الذي يشغل الخدمات التالية ببيئة معزولة محلياً:
- **qablny-api:** الخادم الرئيسي نفسه (.NET 10).
- **postgres:** حاوية قاعدة البيانات الأساسية.
- **redis:** (مخطط لتفعيل الطوابير السريعة والتخزين المؤقت Session cache).
- **livekit:** خادم WebRTC الخاص بمعالجة وتمرير دفق المكالمات المرئية والصوتية بين الأجهزة.
- **minio:** خادم تخزين الملفات والميديا (كبديل محلي آمن ومجاني لـ AWS S3).
- **libretranslate:** خدمة ذكاء اصطناعي مفتوحة المصدر لترجمة الرسائل بشكل آلي ومستقل عن الإنترنت.

## 🚀 تدفق عمل المكالمات الحية (WebRTC Call Flow)
1. **بدء المكالمة (Initiate):** يقوم المتصل (أ) باستدعاء `InitiateCall` في `ChatHub`. يرسل الخادم إشارة `IncomingCall` للمستقبل (ب).
2. **قبول المكالمة (Accept):** يقوم (ب) بالرد واستدعاء `AcceptCall`.
3. **إنشاء الغرفة والتوكن:** يقوم الخادم (بشكل خلفي) بالتواصل مع `LiveKitService` لإنشاء غرفة (Room) فريدة وتوليد توكنات دخول خاصة للمستخدمين (أ) و (ب). يتم إرسال التوكنات للمتصلين عبر SignalR.
4. **تسجيل البداية:** يقوم الـ Hub بتسجيل وقت بداية المكالمة في الذاكرة (ConcurrentDictionary).
5. **الاتصال الفعلي:** يقوم تطبيقي كلا المستخدمين باستخدام `livekitApi` للاتصال بخادم الـ WebRTC والبدء ببث الصوت/الصورة.
6. **إنهاء المكالمة (End):** يستدعي أحدهما `EndCall`. يقوم الخادم بحساب الفارق الزمني (TimeSpan) وحفظ رسالة `Call Log` في الـ DB وإرسالها عبر SignalR للطرفين لتظهر في الشات كمكالمة مسجلة.
